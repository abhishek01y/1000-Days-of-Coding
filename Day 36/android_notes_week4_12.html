Week 4

Permissions & Application Resources

How Android controls access to sensitive features, and how app resources are organized.

Why permissions exist → 3 types of permissions → How resources work → android.R to access them

By design, no Android app has permission to perform any operations that would adversely affect other apps, the OS, or the user. This is a central security principle. Apps must explicitly request permissions, and users must approve sensitive ones.

The purpose of a permission is to protect the privacy of an Android user. Permissions are declared in the Manifest and requested at runtime (Android 6.0+) for dangerous permissions.

4.1Three Types of Permissions

Normal Permission

  • Low risk to user privacy
  • Granted automatically at install
  • No user dialog shown
  • Example: ACCESS_WIFI_STATE
  • Example: SET_WALLPAPER
  • Example: INTERNET

Signature Permission

  • Granted only to apps signed with the same certificate
  • Used between apps from same developer
  • No user dialog needed
  • Example: MANAGE_DOCUMENTS
  • Example: READ_VOICEMAIL
  • Affects third-party apps

Dangerous Permission

  • High risk — accesses private data
  • User must approve at runtime
  • Android 6.0+ requirement
  • Example: RECORD_AUDIO
  • Example: SEND_SMS
  • Example: camera, contacts, location
Important rule — Normal permissions auto-granted If your app lists normal permissions in its manifest, the system automatically grants those permissions. You do NOT need to ask the user. Only dangerous permissions require a runtime request dialog. This is a common MCQ trap.
Android 6.0 (Marshmallow) change Before Android 6.0, all permissions were requested at install time. From Android 6.0 onwards, dangerous permissions must be requested at runtime. The answer for "when is permission requested" = Either Runtime or Install-Time depending on the version.

4.2Application Resources

Resources are the non-code files your app uses — images, layouts, strings, colors, animations. They are stored separately from Java class files in the Android project, under the /res directory.

res/ drawable/ ← images, icons (PNG, JPG, XML drawables) layout/ ← XML layout files for screens values/ ← strings.xml, colors.xml, dimens.xml raw/ ← ← videos, audio files stored here anim/ ← animation XML files xml/ ← arbitrary XML files mipmap/ ← launcher icons

All resources are accessible via the android.R class. The android package contains all kinds of resources which you can browse by looking in the android.R subclasses.

Key rules about resources:

• All resources must be stored under /res in specially named subdirectories that must be lowercase.
• Most common resource types are stored in XML files.
• Android resource files are stored separately from Java class files.
• Resources like bitmaps, layouts, and animations are all examples of app resources.

How to set an image resource dynamically (code) To dynamically assign an image named car from the drawable folder to an ImageView:
image.setImageResource(R.drawable.car) — this is the correct method. Not setImageDrawable, not setImage, not setResource.

User Interface — Layouts, Views & Widgets

How Android screens are built from Views and ViewGroups, and the different layout types available.

View (basic UI block) → ViewGroup (contains Views) → Layout (arranges ViewGroups) → Widgets (interactive controls)

5.1View and ViewGroup

Every UI element in Android is a View. A View occupies a rectangular area on screen and is responsible for drawing and event handling. All interactive UI components (buttons, text fields, checkboxes) are subclasses of View.

A ViewGroup is a special type of View that can contain other Views (and other ViewGroups). ViewGroup is the base class for layouts. The difference: View = leaf node, ViewGroup = container node.

ViewBase class for widgets — used to create interactive UI components (buttons, text fields, etc.). Draws itself and handles events.
ViewGroupBase class for layouts — invisible container that holds other Views and defines how they are arranged.
Layout controlsDetermine how and where on screen child controls are drawn.
All controls available inandroid.widget package
Hide/show a ViewsetVisibility(int) method
Set layout attributes at runtimeTRUE — you CAN set layout attributes programmatically at runtime.

You can create a layout in two ways: using XML resource files (declarative, design-time) or programmatically in Java code (runtime). Both are valid — either XML Resources or Programmatically.

5.2Layout Types

The Android SDK framework includes three main built-in layout types. Each arranges child views differently.

Layout 1

LinearLayout

Organizes child Views in a single row or column, depending on orientation attribute (horizontal or vertical). Designed to block out an area for a single item.

Layout 2

RelativeLayout

Allows you to specify where child view controls are in relation to each other. Views are positioned relative to parent or sibling views.

Layout 3

FlowLayout

Designed to block out an area on screen to display a single item. Less commonly used; wraps content and flows to next line.

Exam trap — layouts built in to Android SDK The answer is FlowLayout, RelativeLayout and LinearLayout — all three are built-in. When the question asks which one arranges in a single row/column = LinearLayout. Which one lets you position relative to each other = RelativeLayout.

Content Providers & Data Storage

The four ways Android apps store data, and how to work with SQLite databases.

4 storage options → Shared Preferences → SQLite Database → Internal / External Storage

6.1Android Storage Options

Android provides four storage options for persisting app data. Each serves a different purpose and access level.

Shared Preferences
Stores primitive data in key-value pairs. Private to the app. Use getSharedPreferences() to access. Best for settings, flags, small values like username. Data is stored in a private XML file.
Internal Storage
Stores data on device memory, private to the app. Not accessible by other users or apps (except the developer). Good for files that must be private.
External Storage
Stores data on shared external storage (SD card or public internal storage). Accessible by other apps and users. Not suitable for private data.
SQLite Databases
Stores structured data in a private database. Full SQL support. Accessible within the app. Database is stored in a private location.
Memory trick — which stores what Shared Preferences = key-value pairs (primitive data)
Internal Storage = private files (not accessible by anyone else)
External Storage = shared files (anyone can read)
SQLite = structured relational data (tables, rows, SQL queries)

6.2SQLite Database

SQLite is an open-source, relational SQL database that stores data in a text file on the device. It does NOT need a server connection — you don't need to establish any kind of connection to access it. It supports all the relational database features.

To use SQLite in Android:

Step 1: Create a class that inherits from SQLiteOpenHelper — this is the first and required step.
Step 2: Override public abstract void onCreate(SQLiteDatabase db) — called once when database is first created.
Step 3: Use the helper to get a readable/writable database instance and run SQL queries.

SQLite typeOpen-source and relational
No connection neededTRUE — SQLite doesn't need a server connection
Supports all relational featuresTRUE — full SQL support (SELECT, INSERT, UPDATE, DELETE)
Database stored inprivate database location
First step to use SQLiteCreate class inheriting from SQLiteOpenHelper
Method called on first DB creationpublic abstract void onCreate(SQLiteDatabase db)
Good adapter for database dataCursorAdapter

VideoViewPlays video files. Found in Widget palette category.

JSON Web Services & Content Providers

How Android apps exchange data over the internet, and how Content Providers share data between apps.

Web services use XML/JSON → JSON format & syntax → AsyncTask for background work → Content Providers share data

Web services allow apps to exchange data over the internet. They generally use XML for data messaging. JSON is a lighter alternative to XML for the same purpose.

7.1JSON — JavaScript Object Notation

JSON stands for JavaScript Object Notation. It is data-oriented (not document-oriented, not tag-oriented). JSON is extended from JavaScript. It is an alternative to XML for data exchange, and for the same data, JSON file size is smaller than XML.

JSON Object — { curly brackets }

  "open": true
}

Key must always be a String
Represents a single object

JSON Array — [ square brackets ]

[
  "Android",
  "iOS",
  "Windows"
]

[ ] = array
{ } = object

7.2AsyncTask — Background Operations

Network calls (fetching JSON from web) cannot be done on the main UI thread — it would freeze the app. AsyncTask allows performing background operations and publishing results on the UI thread without managing threads manually.

AsyncTask best forShort operations (not long-running background tasks)
Must override at leastdoInBackground(Params...) — this is the only required override
doInBackground()Runs on background thread. Do the actual work here (network call, DB query).
onPostExecute(Result)Runs on UI thread after background computation finishes. Update UI here.
onPreExecute()Runs on UI thread before background work starts. Show progress dialog here.
onProgressUpdate()Runs on UI thread to publish progress during background execution.

7.3Content Providers

A Content Provider is a component that supplies data from one application to others on request. It is a standard interface for sharing data between apps. Content Providers are TRUE components of an Android app.

Connection — Week 2 → Week 7 Content Providers were introduced as one of the 4 Android components in Week 2. In Week 7 we go deeper — they are the mechanism that lets different apps share structured data safely through a standard URI-based interface.

Animations in Android apps can be performed through XML or Android code — both methods are valid.

Perform animations viaXML or android code — either method

11.1Android Sensor Framework (ASF)

ASF stands for Android Sensor Framework. It gives access to sensors built into the device. Sensors can be hardware-based (physical chips) or software-based (virtual sensors that mimic hardware).

The software-centric sensors mimic hardware-based sensors — this is TRUE.

Class 1

SensorManager

Use this to make an instance of the sensor administration. Entry point for the sensor system — access all sensors.

Class 2

Sensor

Use this to make an instance of a particular sensor — e.g. accelerometer, gyroscope, light sensor.

Interface

SensorEventListener

Interface to make two callback strategies that get warnings when sensor values change or accuracy changes.

Object

SensorEvent

The object passed to the listener callbacks — contains sensor data values at a specific point in time.

Week 12

Publishing Android Application

The final steps — preparing, signing, and publishing your app to the Google Play Store.

Prepare app → Generate signed APK → Upload to Play Store → Monetize

Publishing is the last phase of the Android application development process. The complete development cycle is: Design → Develop → Test → Publish.

12.1APK Types and Signing

An APK (Android Package) is the file format for distributing Android apps. Android can generate two types of APK: signed and unsigned.

Last phase of developmentPublishing
APK typesSigned and unsigned
Signed APKRequired for Google Play Store submission. Signed with a private key (keystore). This proves the app is from a verified developer.
Unsigned APKFor testing only. Cannot be uploaded to Play Store.
WAP is stripped-down version ofhttp (not https, not ftp)

12.2Publishing Process

Step 1 — Prepare: Remove debug code, check app permissions, optimize images and resources.
Step 2 — Build signed APK: Generate a signed APK using Android Studio (Build → Generate Signed APK). Create or use a keystore file with your private key.
Step 3 — Create Developer Account: Register on Google Play Console (one-time $25 fee).
Step 4 — Upload: Upload the signed APK, add screenshots, description, category, and pricing.
Step 5 — Publish: Submit for review. App goes live after Google review.

Exam Focus — Week 12
  • Last phase of development = publishing
  • APK types = signed and unsigned
  • WAP = stripped-down version of http
  • GestureDetector class used for = detecting common gestures
  • SpeechRecognizer used for = speech to text
