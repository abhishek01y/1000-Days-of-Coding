Week 1

Android Platform, Architecture & Tools

Understanding what Android is, how it is structured internally, and the tools used to build apps.

What is Android? → How is it structured? → How does it run apps? → What tools do we use?

Android is an open-source, Linux-based operating system designed primarily for mobile devices. It was developed by Google and is the world's most widely used mobile platform. Being open-source means anyone can view, modify, and use the code — which is why hundreds of manufacturers use Android in their phones.

Key facts: it is free to use, built on the Linux kernel, and uses Java or Kotlin as primary programming languages. Apps are compiled into APK (Android Package) files before being installed.

Why the Linux base matters Android being Linux-based is not just a trivia fact. It means Android inherits Linux's security model, process management, and hardware driver system. Every question about Android's foundation traces back to this.

1.1Android Architecture — The 5 Layers

Android is organized in 5 layers stacked on top of each other. Each layer depends on the one below it. This layered design is why Android can run on thousands of different devices — each device only needs to change the bottom layers, not the top ones.

5Applications— Gmail, Camera, Maps, your own apps

The Application Framework (Layer 4) is what you actually use when writing Android code — the APIs for Activities, Services, Intents, Permissions etc. all live here. As a developer, you rarely touch Layers 1–3 directly.

1.2ART — Android Runtime

When you write Android code in Java, it goes through this pipeline before running on a device:

Java source code (.java) → compiled to Java bytecode (.class files) → converted to DEX format (.dex files) ← compact bytecode for mobile → ART executes the .dex files on the device

ART (Android Runtime) is the engine that runs your compiled app code. It replaced the older Dalvik runtime (used until Android 4.4). ART is significantly faster because of how it compiles code.

ART has three core features that work together:

AOT — Ahead-of-Time compilationCompiles app code before it runs — at install time. Makes apps launch faster because work is done upfront.
JIT — Just-in-Time compilationCompiles code during execution — only the parts currently being used. Re-introduced in Android 7.0 (Nougat) to work alongside AOT.
GC — Garbage CollectionAutomatically frees memory that is no longer needed by the app. Prevents memory leaks without the developer manually managing memory.
How AOT + JIT work together (Android 9.0+) On Android 9.0 and higher, both run simultaneously. AOT compiles commonly-used code at install time. JIT handles less-used code at runtime and feeds usage data back to improve future AOT compilations. This is called profile-guided compilation and is why DEX-to-machine-code conversion starts at 9.0.

1.4Development Tools

Android StudioOfficial IDE for Android development. Built on IntelliJ IDEA by JetBrains, adopted by Google. Replaced Eclipse in 2014 as the official tool. Includes code editor, layout designer, emulator, debugger, and Gradle build system.
APK AnalyzerBuilt into Android Studio. Used to inspect the contents of an APK file — view DEX files, resources, manifest structure, and file size breakdown.
AVD (Android Virtual Device)An emulator configuration tool. Creates a software simulation of an Android device by defining both Hardware (screen size, RAM, camera) AND Software (Android version, locale) options.

GradleBuild automation system. Compiles your Java/Kotlin code, processes resources, and packages everything into a final APK. Configured via build.gradle files.

Week 2

Android App Structure — Components, Manifest & Lifecycle

How an Android app is organized, what must be declared, and how Activity states work.

4 Components → Manifest declares them all → Activity has a lifecycle → Permissions protect sensitive data

An Android app is not a single monolithic program. It is a collection of loosely coupled components, each serving a specific purpose. There are exactly 4 types of components, and every component that exists in your app must be declared in the AndroidManifest.xml file.

2.1The 4 App Components

Component 1

Activity

Represents a single screen with a UI. Every screen the user sees is one Activity. Most apps have multiple Activities. The Activity provides the window in which the app draws its interface.

Component 2

Service

Runs in the background with no user interface. Used for long-running operations — music playback, file downloads, syncing. To create one, extend the Service base class.

Component 3

Broadcast Receiver

Listens for and responds to system-wide broadcast messages. Examples: battery low, screen turned off, incoming SMS, network changed. Has no UI but can display a notification.

Component 4

Content Provider

Manages a shared set of app data that other apps can access. Example: the system Contacts app exposes contacts via a Content Provider. Data can live in SQLite, files, or on the web.

Connecting Week 2 → Week 3 Three of these four components — Activity, Service, and Broadcast Receiver — are activated by Intents. Content Providers are activated separately via ContentResolver. This is why Intents (Week 3) are the central messaging system in Android — they activate three of the four building blocks.

2.2The AndroidManifest.xml File

Every Android app must have an AndroidManifest.xml file at the root of its project. Before Android starts any component of your app, it reads this file first. If a component is not declared in the Manifest, Android does not know it exists and cannot start it.

The Manifest has three main jobs:

1. Declares all four types of components the app contains.
2. Declares what permissions the app needs (camera, internet, contacts, etc.).
3. Declares minimum API level and hardware requirements.

<manifest> <!-- REQUIRED (1 of 2 required elements) --> <application> <!-- REQUIRED (2 of 2 required elements) --> <activity android:name=".MainActivity"/> <!-- child of application --> <service android:name=".MyService"/> <receiver android:name=".MyReceiver"/> <provider android:name=".MyProvider"/> </application> <uses-permission android:name="android.permission.CAMERA"/> </manifest>
Critical rule — only 2 elements are required Only <manifest> and <application> are required. All others are optional in structure, though practically necessary if you use those components. Activity elements are declared as children of <application>, not directly under <manifest>.

2.4Permissions

Android protects sensitive user data and device features with a permission system. An app must explicitly declare which permissions it needs and, for dangerous permissions (camera, contacts, SMS, location), must request approval from the user at runtime.

Normal permissions (like internet access) are granted automatically at install. Dangerous permissions require a user dialog. All permissions — regardless of type — must be declared in the Manifest first using <uses-permission>.

Manifest + Permissions connection Permissions declared in Manifest = what the app can ask for. Runtime approval = whether the user grants it. Both are needed. Missing the Manifest declaration means the app cannot use that feature even if the user would approve.

Week 3 🔥

Intents — Inter-Component Communication

How Android components communicate with each other, and how the system finds the right component to handle a request.

What is an Intent? → Explicit vs Implicit → Intent Attributes → Intent Resolution → Intent Filter

3.1What is an Intent?

An Intent is a messaging object that requests an action from another app component. It provides late runtime binding between components — meaning the connection between the sender and the receiver is made at runtime, not at compile time. This is what allows different apps to work together without knowing about each other in advance.

Intents have three primary uses in Android:

1. Starting an Activity → call startActivity(intent)
2. Starting a Service → call startService(intent)
3. Delivering a Broadcast → call sendBroadcast(intent)

Week 2 → Week 3 Connection In Week 2 you learned the 4 components. Three of them — Activity, Service, Broadcast Receiver — are all started by Intents. The Intent is the mechanism that activates them. Content Provider is the only component not activated by an Intent.

3.2Explicit vs Implicit Intents

Explicit Intent

  • Specifies the exact component (class name) to start
  • Used for components within your own app
  • Android doesn't need to search — destination is fixed
  • Example: open SettingsActivity from MainActivity
  • Component attribute is set → makes it explicit

Implicit Intent

  • Does not name a specific component
  • Declares a general action to perform
  • Android searches all apps for one that can handle it
  • Example: ACTION_DIAL → any phone app can handle it
  • Requires Intent Filter in target to receive
Key distinction Explicit = you decide which component runs (you name it). Implicit = Android decides which component runs (based on action + data). This is the fundamental difference between the two types.

3.3Intent Attributes

An Intent carries information Android uses to determine which component should handle it. The attributes are divided into one primary (mandatory) and four secondary (optional):

Action
PRIMARY
Only mandatory attribute
Secondary Attributes — 4 total
Category Type Component Extras
Action (Primary)A string naming the action to perform. This is the only primary/mandatory attribute. Examples: ACTION_VIEW, ACTION_DIAL, ACTION_SEND, ACTION_MAIN.
Category (Secondary)Additional information about the type of component that should handle the Intent. Example: CATEGORY_LAUNCHER = show in app launcher. CATEGORY_DEFAULT = can respond to implicit intents.
Type (Secondary)Specifies an explicit MIME type of the Intent data. Example: "image/jpeg", "text/plain", "video/mp4". This overrides the type that would normally be inferred from the data URI.
Component (Secondary)Specifies the exact class name of the component to start. When this is set, the Intent becomes Explicit. All other attributes become optional.
Extras (Secondary)Key-value pairs of additional data passed to the target component. Example: passing a username string or an order ID to the next Activity using putExtra().
DataA URI pointing to the data to act upon. Example: tel:123 for a phone number, http://example.com for a URL, content://contacts/1 for a contact record.

3.4Intent Resolution

Intent resolution is the process Android uses to find which component should handle an implicit Intent. It compares the Intent against the Intent Filters of all installed apps and finds matches. This happens at runtime when you call startActivity().

Android uses exactly three pieces of information from the Intent for resolution:

App sends an implicit Intent
↓ Android reads these 3 pieces
1. Action 2. Data (URI + MIME type) 3. Category
↓ compared against Intent Filters of installed apps
matched component can be →
Activity Service BroadcastReceiver

If one match is found → Android starts it directly. If multiple matches → Android shows a chooser dialog. If no match → app crashes with ActivityNotFoundException.

Explicit Intents skip resolution entirely Intent resolution only applies to implicit Intents. When you send an explicit Intent (with component name set), Android delivers it directly — no searching, no Intent Filters checked.

3.5Intent Filter

An Intent Filter is declared in AndroidManifest.xml inside a component declaration. It specifies what types of implicit Intents a component is willing to receive. It is the component's way of advertising its capabilities.

<activity android:name=".ShareActivity"> <intent-filter> <action android:name="android.intent.action.SEND" /> <category android:name="android.intent.category.DEFAULT" /> <data android:mimeType="text/plain" /> </intent-filter> </activity>

This says: "ShareActivity can handle an ACTION_SEND Intent with plain text data." Any app that creates such an Intent may trigger this Activity. Intent Filters give components the ability to launch from both explicit AND implicit requests.

Intent Filter vs Explicit Intent Intent Filters are only checked for implicit Intents. For explicit Intents (where the component is named directly), Android ignores all Intent Filters and delivers directly to the named component.
